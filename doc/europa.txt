1. Overview						*vimcmdline_overview*

europa runs your code like a Jupyter notebook inside Neovim: split a file into
"# %%" cells, run them through a headless Jupyter kernel, and see each cell's
text output rendered inline under the cell (see |vimcmdline-notebook|). Plots
are drawn INLINE in the cell output by europa itself via the kitty graphics
protocol (on a capable terminal — kitty, ghostty), or in the plotty package's
tmux pane on other terminals (see |cmdline_notebook_figures|).

europa is also a plain send-to-interpreter plugin, built on vimcmdline: it
sends lines, paragraphs, blocks, and files from either Vim or Neovim to a REPL
for Clojure, Golang, Haskell, JavaScript, Julia, Jupyter, Lisp, Macaulay2,
Matlab, Prolog, Python, Ruby, Sage, Scala, Shell script, and Swift.

The interpreter may run in a Neovim built-in terminal (a Neovim buffer), an
external terminal emulator, or in a tmux pane. When running in either a Neovim
built-in terminal or an external terminal, the plugin runs one instance of the
REPL application for each file type. When running in a tmux pane, it runs one
REPL application per Vim instance.

europa is a fork of vimcmdline; it keeps the cmdline_* settings and mappings,
so existing vimcmdline configurations keep working.

==============================================================================
2. Usage						*vimcmdline_usage*

If you are editing one of the supported file types, in Normal mode press |,s|
to start the interpreter, then use the mappings described in
|vimcmdline_mappings| to send code to it. For notebook mode (Python, Neovim),
see |vimcmdline-notebook|.

3. Mappings						*vimcmdline_mappings*

The default mappings are listed below. The default prefix is "," ; set
|cmdline_default_keybindings| to 1 to use the original "<LocalLeader>" prefix.
Each mapping can also be changed individually (see |vimcmdline_options|).

Normal mode: >
	,s			Start the interpreter
	<Space>			Send the current line and move to the next
	,<Space>		Send the current line and keep the cursor
	,q			Send the quit command to the interpreter
<
For languages that can source chunks of code:

Visual mode: >
	<Space>			Send the selected text
<
Normal mode: >
	,f			Send the entire file
	,p			Send from the line to the end of paragraph
	,b			Send block between the two closest marks
<
Code-block (cell) mappings, see |vimcmdline_blocks|: >
	,c			Execute the current code block
	,n			Execute the current code block and jump to next
	,e			Execute from the current line to the block end
	,]			Jump to the next block
	,[			Jump to the previous block
<
These are also available as commands and <Plug> mappings, for binding your
own keys without a raw ":call Func()<CR>": >
	:CmdLineExecCell            <Plug>(cmdline-exec-cell)
	:CmdLineExecCellJumpNext    <Plug>(cmdline-exec-cell-jump-next)
	:CmdLineExecAllCells        <Plug>(cmdline-exec-all-cells)
	:CmdLineExecAllCellsBelow   <Plug>(cmdline-exec-all-cells-below)
	:CmdLineExecToEnd           <Plug>(cmdline-exec-to-end)
	:CmdLineNextCell            <Plug>(cmdline-next-cell)
	:CmdLinePrevCell            <Plug>(cmdline-prev-cell)
<
ExecAllCells runs every cell top to bottom; ExecAllCellsBelow runs the cell
under the cursor and every cell below it. Both skip whitespace-only cells and
restore the cursor when done. Neither has a default key mapping: bind the
<Plug> mappings above yourself if you want one.

A count prefix (e.g. "3<Plug>(cmdline-next-cell)") moves/executes N cells for
ExecCellJumpNext/NextCell/PrevCell. ExecCell/ExecToEnd/ExecAllCells/
ExecAllCellsBelow always run once regardless of a count, since a count
re-running the same cells N times would just duplicate their side effects
rather than doing anything useful.
Notebook-mode mappings, see |vimcmdline-notebook| (only when enabled): >
	,k			Toggle notebook mode (start/stop the kernel)
	,K			Clear the output under the current cell
	,o			Open the current cell's full output in a popup
	,i			Interrupt the running cell (stop a runaway loop)
<
==============================================================================
4. Code blocks						*vimcmdline_blocks*

A file can be divided into code blocks (also called "cells") delimited by a
separator line. The separator defaults to "# %%", matching the convention used
by Jupyter and VSCode, and is configurable with |cmdline_block_sep|.

The block mappings (|vimcmdline_mappings|) are defined only for file types that
provide a function to source a chunk of code (for example Python, Julia,
Shell). The separator is matched literally, so special characters in it are not
treated as a regular expression.

When the cursor is on a separator line, the current block is the region that
begins just after that separator.

6. Colors						*vimcmdline_colors*

On Neovim, the interpreter output can be colorized. The colors are set with a
family of cmdline_color_* options, each of which accepts the hex code of a
foreground color, the ANSI number of a foreground color, or a complete
highlighting specification. To follow the current |colorscheme| instead, set: >
	let cmdline_follow_colorscheme = 1
<
See the README for the full list of cmdline_color_* options and examples.

7. Notebook mode					*vimcmdline-notebook*
							*vimcmdline_notebook*

Notebook mode is ON by default (Neovim only, Python only). Each code block
(|vimcmdline_blocks|) is run by a headless Jupyter (ipykernel) kernel and its
TEXT output is shown inline, directly under the cell, using virtual lines.
Matplotlib AND plotly figures are drawn INLINE in the same output box by
default (kitty graphics; survives tmux and SSH); alternatively they can render
in the plotty package's own tmux pane (https://github.com/xuesoso/plotty) — see
|cmdline_notebook_figures|. Plotly needs kaleido + nbformat importable in the
kernel's Python (both ship with a normal Jupyter install); without them europa
leaves plotly's renderer untouched and matplotlib is unaffected.

The mode is on by default; set cmdline_notebook_enable = 0 for classic
REPL-only behavior. Run :checkhealth europa to verify prerequisites.

Requirements:
  - Neovim 0.10+.
  - Python with jupyter_client and ipykernel:
      pip install jupyter_client ipykernel
  - For inline figures (the default): a terminal that implements
    kitty-graphics VIRTUAL PLACEMENTS — kitty or ghostty — plus
    'termguicolors', and inside tmux (>= 3.3) `set -g allow-passthrough on`.
  - For the plotty pane: pip install plotty, and run inside tmux (>= 3.4) in
    a sixel/kitty-capable terminal.

NOTE: inline display uses a specific SUBSET of the kitty graphics protocol
(virtual placements with Unicode placeholders — what lets figures live in
virtual lines and survive scrolling/tmux), which far fewer terminals implement
than the base protocol. kitty and ghostty support it; WezTerm, Konsole, iTerm2
and other partial implementations do NOT. europa detects the terminal at
kernel start — inside tmux it asks tmux for the ATTACHED CLIENT's terminal
(#{client_termname}); otherwise it checks $KITTY_WINDOW_ID / $GHOSTTY_* and
$TERM (which ssh forwards) — and on an incapable terminal figures FALL BACK
automatically: to the plotty pane when inside tmux with plotty importable by
the kernel's python, else to a "[figure not displayed: ...]" text note, never
raw escape data. Nested tmux carries images only over sixel, so it needs the
plotty pane too. Overrides: |cmdline_notebook_kitty_terms| teaches the
detection about terminals it cannot know (it REPLACES the name list), and an
explicit >
	let cmdline_notebook_figures = 'inline'
<skips the detection entirely; set it to 'plotty' to always use the pane
(this was the default before 'inline' became the default).

It is on by default; run a cell and the kernel starts automatically. To turn
it off (classic REPL only), set in your |vimrc|: >
	let cmdline_notebook_enable = 0
<
Toggle it per buffer with |<LocalLeader>k| (or :CmdLineNotebookToggle). While
on, the cell mappings (|vimcmdline_mappings|) and the send mappings run
through the kernel and render output inline instead of to a REPL.

You do not have to toggle notebook mode on by hand: any cell-exec command
(ExecCell, ExecCellJumpNext, ExecAllCells, ExecAllCellsBelow) auto-enables it
and starts the kernel when the feature is on (|cmdline_notebook_enable|), the
filetype has an interpreter, and no classic REPL is already running for it (a
live REPL wins). If notebook mode is already on but no kernel is attached
(never started, stopped, or crashed), the same commands start one before
sending. Cells submitted while the kernel is still booting are queued and run
once it is ready. Repeated exec commands (including key-repeat) never launch a
second kernel.

Commands: >
	:CmdLineNotebookToggle      toggle notebook mode (start/stop kernel)
	:CmdLineNotebookStart       start the kernel
	:CmdLineNotebookStop        stop the kernel
	:CmdLineNotebookRestart     restart the kernel (clears output)
	:CmdLineNotebookInterrupt   interrupt the running cell
	:CmdLineNotebookClear       clear output under the current cell
	:CmdLineNotebookClearAll    clear all output in the buffer
	:CmdLineNotebookOpenOutput  open the current cell's full output in a
			popup; a cell's inline figure is shown LARGER there
			(~85% of the editor width, freed when the popup closes)
	:CmdLineNotebookCollapse    toggle the presentation view (see below)
<
						*:CmdLineNotebookCollapse*
:CmdLineNotebookCollapse (default mapping: ,z or <LocalLeader>z)
			Toggle a PRESENTATION VIEW: every non-markdown cell's
			code collapses to a single fold line — the "# %%"
			separator with its title and a hidden-line count —
			while "# %% [markdown]" cells and the rendered inline
			outputs stay visible, so the buffer reads like a
			report. The lines anchoring output boxes stay
			unfolded wherever they actually are (an output is
			anchored to the line its cell ended on when it RAN,
			which drifts with edits — a closed fold would hide
			it); cells with no output fold entirely. The view is
			LIVE ('foldexpr'-based): edits track, and newly added
			cells fold themselves. While it is active the
			window's usual fold method is displaced (one
			'foldmethod' per window); toggling off restores the
			saved fold settings ('foldmethod', 'foldexpr',
			'foldtext', 'foldenable', 'foldminlines',
			'foldlevel') — manual folds the window had are
			cleared. The view is strictly PER-SESSION: if a
			:mkview/:loadview setup (or a session file) captured
			the collapsed state, it is recognized by its
			'foldexpr' signature and dissolved back to your
			global fold options the next time the file is shown —
			your own folding (e.g. indent folds of functions) is
			what persists, never the presentation view.
Options:
							*cmdline_notebook_enable*
cmdline_notebook_enable   Master switch for notebook mode (Neovim). Default 1
			(on). When 0, none of the commands, mappings, or
			autocommands are defined and behavior is classic REPL
			only.

							*cmdline_notebook_python*
cmdline_notebook_python   Python executable that runs the kernel bridge (must
			have jupyter_client + ipykernel). Default: 'python3'.

							*cmdline_notebook_kernel_name*
cmdline_notebook_kernel_name  Jupyter kernelspec name. Default: 'python3'.

							*cmdline_notebook_plotty*
cmdline_notebook_plotty   LEGACY (pre-|cmdline_notebook_figures|). Only read
			when |cmdline_notebook_figures| is unset: 1 routes
			figures to the plotty pane, 0 turns figure display
			off. Unset by default. Prefer setting
			|cmdline_notebook_figures| directly.

							*cmdline_notebook_startup_code*
cmdline_notebook_startup_code  List of extra Python lines run once at kernel
			start, e.g. to enable plotty with custom options.

							*cmdline_notebook_max_lines*
cmdline_notebook_max_lines  Maximum inline output lines shown per cell before
			truncation. Default: 20. Use :CmdLineNotebookOpenOutput
			to see the full retained output.

							*cmdline_notebook_max_kept_lines*
cmdline_notebook_max_kept_lines  Retention cap per cell: at most this many
			output lines are KEPT in memory — the first and last
			halves, with an exact "··· N lines elided ···" marker
			between — so a runaway cell (`while True: print(...)`)
			cannot grow memory or redraw cost without bound.
			Figures are never elided. A one-time warning suggests
			:CmdLineNotebookInterrupt when a cell overflows.
			Applies to newly executed cells. Default: 10000;
			0 = keep everything (the pre-retention behavior).
			NOTE: :CmdLineNotebookOpenOutput shows the retained
			output — with the middle elided once a cell overflows.

							*cmdline_notebook_kernel_timeout*
cmdline_notebook_kernel_timeout  Seconds to wait for the kernel. Default: 30.

							*cmdline_notebook_border*
cmdline_notebook_border   Border drawn around inline output: 'rounded'
			(default), 'single', 'double', or 'none'. Drawn with
			the CmdlineNotebookBorder highlight group.

							*cmdline_notebook_border_color*
cmdline_notebook_border_color  Color of the output border. Accepts a #rrggbb
			hex, a cterm color number, or a full :highlight
			argument string (like the cmdline_color_* options).
			Defaults to dark blue (#005faf / cterm 25).

							*cmdline_notebook_statusline*
cmdline_notebook_statusline  When 1 (default), show a kernel-status segment in
			'statusline' and, when present, in vim-airline:
			" ⏳ kernel" while starting, " ● kernel" when ready,
			" ⟳ running +N" while executing (N = cells queued behind
			the running one), empty otherwise. Set to 0 to place the
			VimCmdLineNotebookStatus() segment yourself. For lualine,
			use a component calling VimCmdLineNotebookStatus().

							*cmdline_notebook_airline_section*
cmdline_notebook_airline_section  vim-airline section the kernel status is added
			to (a letter 'a'..'z'). Default: 'x'.

							*cmdline_notebook_output_win*
cmdline_notebook_output_win  Window used by :CmdLineNotebookOpenOutput: 'float'
			(a read-only centered popup; q or <Esc> closes it) or
			'split'. Default: 'float'.

							*cmdline_notebook_figures*
cmdline_notebook_figures  Where matplotlib figures render: 'inline' (kitty
			graphics drawn inside the cell's output box — needs a
			terminal with kitty-graphics virtual placement support
			such as kitty/ghostty, 'termguicolors', and tmux
			`allow-passthrough on` when inside tmux), 'plotty'
			(plotty's tmux pane), or 'none'. UNSET by default;
			'inline' is then the resolved default, subject to
			terminal detection with automatic fallback to the
			plotty pane (inside tmux, plotty installed) or a text
			note — see |vimcmdline-notebook|. Setting 'inline'
			EXPLICITLY skips the detection (a deliberate override
			for terminals it cannot see). When this option is
			unset, an explicit legacy g:cmdline_notebook_plotty
			still wins (1 => 'plotty', 0 => 'none'). Nested tmux
			cannot carry inline images: europa detects it, warns
			at kernel start, and falls back (use 'plotty' there).

						*cmdline_notebook_kitty_terms*
cmdline_notebook_kitty_terms  List of terminal-name substrings the inline
			figure detection treats as kitty-graphics capable,
			matched against $TERM (or tmux's #{client_termname}
			inside tmux). Default: ['kitty', 'ghostty']. Setting
			it REPLACES the list — include the defaults if you
			still want them. For terminals that ship virtual
			placement support the default list cannot know
			about: >
	let cmdline_notebook_kitty_terms = ['kitty', 'ghostty', 'myterm']
<

							*cmdline_notebook_figure_size*
cmdline_notebook_figure_size  Inline figure width in terminal columns, capped
			to the window width. Default: 50. Applies LIVE:
			changing it (or :CmdLineNotebookFigureSize) re-renders
			figures already on screen; text output is untouched.

							*cmdline_notebook_figure_rows*
cmdline_notebook_figure_rows  Explicit inline figure height in rows; 0
			(default) derives the height from the image's aspect
			ratio. Applies live, like the width.

							*:CmdLineNotebookFigureSize*
:CmdLineNotebookFigureSize {width} [{height}]
			Set the inline figure width (columns) and optionally
			height (rows; omitted = keep aspect), then re-transmit
			and redraw every figure currently on screen.

							*:CmdLineNotebookFigureRefresh*
:CmdLineNotebookFigureRefresh
			Re-transmit every retained figure at its current size.
			Terminals cap their graphics memory (kitty ~320MB) and
			evict the oldest images, leaving blank rectangles; the
			PNGs are retained editor-side, so this restores them.
			Figures are sent in cursor-priority order (other
			buffers first, current buffer nearest-to-cursor LAST):
			if the retained set exceeds the terminal's quota, the
			figures around the cursor survive and far-away ones
			blank out — no command can make the terminal hold more
			than its quota; reduce figure dpi/size or clear old
			cells to fit everything.

							*cmdline_notebook_figure_dpi*
cmdline_notebook_figure_dpi  Source resolution the kernel renders inline figures
			at. This is matplotlib's dpi AND plotly's render scale
			(dpi / 100, matplotlib's baseline dpi), so one knob
			drives both backends: the default 200 renders plotly at
			2x its native 700x500. On-screen size is set separately
			by |cmdline_notebook_figure_size|. Default: 200.

							*cmdline_notebook_figure_cell_aspect*
cmdline_notebook_figure_cell_aspect  Terminal cell height/width ratio used to
			compute the inline figure's rows from its pixel
			aspect. Default: 2.0.

							*cmdline_notebook_exec_marker*
cmdline_notebook_exec_marker  Where to mark each executed cell with its
			execution count N and run status. Values:

			'left' (default)~
			Drawn entirely in the 'signcolumn' instead, costing
			no vertical space and never shifting any text: a
			colored bar spans the cell (separator line included),
			and the separator line's sign is the execution count
			itself — the COLOR is the status: green on success,
			red on failure, a yellow "●" while queued/running,
			"✗" for an aborted cell (which has no count). Counts
			past 99 display as "++" (a sign is at most 2 cells).
			The below-cell rule line and border marker are
			suppressed. Needs 'signcolumn' "auto" (the default)
			or "yes"; the marks track buffer edits and are
			removed when the cell reruns or its output is
			cleared.

			The signs use priority 9 — one below the |:sign-place|
			default (10) used by bookmark/marks plugins — so with
			a 1-slot 'signcolumn' your own signs win the line,
			and with >
				set signcolumn=auto:2
<			bookmarks and the bar display side by side. Colors
			come from these groups (fg-only by default so the bar
			is not a solid block):
				CmdlineNotebookGutterOk   → DiagnosticOk
				CmdlineNotebookGutterErr  → DiagnosticError
				CmdlineNotebookGutterRun  → DiagnosticWarn

			1 or 'below'~
			The classic style: "✓ [N]" ("✗ [N]" on error)
			embedded in the output box's top border, or a single
			rule line ("─── ✓ [N] ───") for cells with no output,
			so they still show that they ran.

			0~
			No marker at all.

8. Adding support for a new language			*vimcmdline_newlang*

  1. Copy the script in the ftplugin directory whose language is closest to
     the one you want to support, and save it as "filetype_cmdline.vim" where
     "filetype" is the output of `echo &filetype` for that language.
  2. Edit the new script and change the values of its variables.
  3. Copy the closest script in the syntax directory and save it as
     "cmdlineoutput_app.vim" where "app" is the interpreter name.
  4. Edit the patterns used to recognize the input line and errors.

 vim:tw=78:ts=8:noet:ft=help:norl:
